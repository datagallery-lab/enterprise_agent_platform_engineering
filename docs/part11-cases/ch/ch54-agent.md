# 第54章 业务案例准入与写作方法

---

## 本章摘要

企业级 Agent 的案例很容易写成“功能演示加效果口号”。材料不够时，作者往往把一次离线实验写成生产效果，或用虚构公司补上组织背景。这样的案例看起来完整，却会误导读者对能力成熟度和风险控制状态的判断。业务案例进入正文前，必须先说明真实任务，再拿出数据、工具、权限、评估和人工确认的证据，并回扣前文的平台能力。中文第一版先建立案例写作的证据门槛，不用想象案例填补落地章节。

## 关键词

案例准入、工程证据、脱敏、案例结构、能力对应

## 学习目标

- 能说明第一版为什么不直接编写案例，以及准入标准如何界定。
- 能识别哪类材料可以支撑一篇合规的案例（真实数据、脱敏处理、工程路径）。
- 能用推荐结构撰写一篇可纳入本书的案例。
- 能把案例结论与前文平台能力章节建立明确对应关系。

---

## 场景引入

企业知识助手、工单/客服 Agent、研发 Agent 和运维 Agent 都适合说明平台能力，也最容易被写成泛泛的功能故事。中文第一版暂不编写具体公司故事，也不虚构上线结果、指标收益或组织背景。案例进入正式正文前，至少要回扣前文的平台能力，并说明数据、工具、权限、评估和人机协同怎样共同支撑一个业务任务。

---

## 54.1 案例纳入标准

案例进入正文之前，要回答一个朴素问题：读者沿着材料，是否复原得出这件事是怎样做成的。如果材料只写“用了 Agent 提效”“接入知识库”“自动生成报告”，却没有任务输入、系统边界、失败处理和评估证据，那么它还不是案例，只是宣传素材。

一个合格案例至少要满足四个条件。

*表54-1：案例纳入正式正文的四个条件与验收口径。来源：本书整理。*

| 条件 | 验收口径 |
| --- | --- |
| 业务任务明确 | 能写清楚用户角色、输入、输出、决策动作和风险边界 |
| 平台能力可对应 | 能映射到 Runtime、Tool Registry、MCP、Planner、Memory、RAG、Eval、Trace 或 Guardrails |
| 数据与工具拓扑清楚 | 能说明涉及哪些系统、数据对象、权限、审计和失败恢复动作 |
| 不夸大效果 | 不写未验证的降本、提效、准确率或上线规模 |

这四个条件不是形式审查。它们分别拦住四类常见问题：任务不清、能力悬空、链路缺证据、效果被夸大。案例作者先按表 54-1 自检，审稿者再按同一口径复核。任何一项说不清，材料只能进入素材库，暂不进入正式正文。

### 54.1.1 任务要能被复原

任务明确，不等于写一句“销售需要自动报价”或“运营需要分析下滑原因”。可复原的任务至少包含五个要素：谁发起、带着什么输入、希望得到什么输出、系统允许做哪些动作、哪些动作必须停下来交给人。

以报价场景为例，“生成报价建议”仍然太粗。更合格的写法是：销售在 CRM 中选择客户和产品包，系统读取历史合同、库存状态和折扣策略，生成报价草稿与风险提示；正式报价、折扣例外和客户承诺由销售经理确认。这样读者才能判断它是 Copilot、Workflow，还是具有 Agent 闭环的任务执行系统。

### 54.1.2 全链路证据覆盖

很多案例会展示一张漂亮的结果截图，却跳过系统如何得到这个结果。对本书来说，截图只能证明“界面出现过”，不足以证明链路可信。案例证据要覆盖输入、检索、工具调用、权限判断、输出、复核和反馈。Trace、Tool Call 记录、评测样本、脱敏日志和人工复核表，往往比最终截图更重要。

*表54-2：案例证据材料与可支持的写作结论。来源：本书整理。*

| 证据材料 | 可以支持的结论 | 不能支持的结论 |
| --- | --- | --- |
| 脱敏任务样本 | 任务类型、输入复杂度、用户角色 | 上线规模、业务收益 |
| Tool Call 记录 | 调用了哪些系统、是否产生副作用 | 工具返回结果一定正确 |
| Trace 与状态机 | 任务如何推进、在哪里等待人工确认 | 模型能力优于其他方案 |
| 评测集与人工复核 | 质量边界、常见失败类型 | ROI、组织效率提升 |
| 脱敏界面截图 | 用户如何看到结果和证据 | 生产链路已经稳定运行 |

写作时要控制证据的外推范围。一次离线评测适合说明质量趋势，不足以说明上线收益；一段脱敏日志适合说明链路形态，不能证明系统已达到生产 SLO。

### 54.1.3 脱敏范围与因果关系保留

案例脱敏不能停留在把公司名改成“某企业”。需要处理的对象包括业务身份、客户信息、系统域名、字段口径、审批角色、工单编号、日志时间和截图中的元数据。对于 Agent 案例，还要特别注意工具调用参数和检索片段，因为它们常常暴露内部系统结构。

*表54-3：Agent 案例脱敏检查项。来源：本书整理。*

| 对象 | 常见泄露点 | 写作处理方式 |
| --- | --- | --- |
| 业务实体 | 客户名、供应商名、门店、合同编号 | 改为角色或类别，保留任务关系 |
| 数据字段 | 内部字段名、指标口径、权限标签 | 保留语义，替换实现细节 |
| 工具调用 | 内网域名、真实 API、账号 ID | 写成接口类型和必要参数 |
| 日志与 Trace | 时间戳、请求 ID、用户 ID | 使用合成 ID，保留状态转移 |
| 截图 | 水印、导航栏、组织架构、头像 | 重新绘制或裁剪，只保留说明所需区域 |

如果脱敏会破坏案例的关键因果关系，就不要强行发布。更稳妥的做法是把材料沉淀为内部复盘，等匿名化样本足够后再进入公开版本。

## 54.2 推荐案例结构

每个案例建议按同一结构展开，便于与前文能力章节互相引用。统一结构的目的，是让读者比较不同场景：哪些能力来自业务差异，哪些能力应该沉淀为平台共性。

1. 业务任务：谁在什么场景下发起任务，为什么固定流程或普通问答不够用。
2. 数据与工具：任务需要访问哪些系统、文档、表、API 或人工审批节点。
3. Agent 设计：如何拆分 Planner、工具调用、记忆、检索、人工确认和结果交付。
4. 运行链路：一次任务从输入到输出经历哪些状态，在哪里记录 trace 和审计证据。
5. 风险控制：哪些动作需要拒绝、降级、复核或转人工。
6. 评估方式：如何用离线样本、人工复核、线上反馈和业务指标判断是否可用。

这六部分的顺序很重要。先写业务任务，避免技术名词先行；再写数据与工具，让读者看到系统碰到的真实边界；随后写 Agent 设计和运行链路，解释 Planner、工具调用、Memory、RAG、人工确认和 Trace 在任务中的分工；风险控制和评估放在后半段，用来避免案例停在“系统能跑”的层面。评估放得过早，案例容易变成指标宣传；平台能力放得过早，案例又会变成架构目录复述。

每一部分都应该经得起审稿追问。业务任务要说明“用户为什么不能用固定流程解决”；数据与工具要说明“哪些字段只能读、哪些动作会产生副作用”；Agent 设计要说明“为什么需要规划而不是一次问答”；运行链路要说明“Trace 怎样还原当时发生了什么”；风险控制要说明“系统在哪些条件下停下”；评估方式要说明“样本和复核口径是否足以支撑结论”。这些问题答不上来，案例就不该进入正文。

### 54.2.1 任务优先的叙述顺序

差的案例常常这样开头：“本案例基于 RAG、Planner、Tool Registry 和 Guardrails 构建了一个智能客服 Agent。”这句话看似完整，其实没有告诉读者为什么需要这个系统。更好的写法是先写任务摩擦：客服需要在几分钟内处理退款争议，但订单系统、物流系统和政策知识库分散在不同系统；固定流程只能覆盖标准退款，遇到促销叠加、部分发货和人工承诺时就会失效。

任务写清楚以后，平台能力才有位置。RAG 用来找政策，Tool Registry 用来约束订单查询和退款试算，Human-in-the-loop 用来处理例外退款，Trace 用来保留证据。这种顺序更接近参考书的行文方式：先让读者看到问题，再给出工程结构。

### 54.2.2 写清楚“不自动化”的部分

Agent 案例容易把自动化写得过满。成熟的企业案例往往会清楚标出系统不做什么。比如销售报价案例中，Agent 可以汇总信息、生成草稿、检查折扣策略，但不能替销售承诺客户；法务审阅案例中，Agent 可以抽取条款和标注风险，但不能给出最终法律意见；财务分析案例中，Agent 可以解释指标变化，但不能替负责人确认披露口径。

这些边界是生产系统可信的前提。写作时应把“自动完成”“需要确认”“拒绝执行”分开列出，读者才能判断案例是否可落地。

*表54-4：案例写作中的自动化边界表达。来源：本书整理。*

| 动作类别 | 写作时应说明 | 示例 |
| --- | --- | --- |
| 自动完成 | 输入、输出、成功条件和审计记录 | 查询历史订单并生成退款原因摘要 |
| 需要确认 | 谁确认、确认什么、超时如何处理 | 经理确认超额折扣后才生成正式报价 |
| 拒绝执行 | 拒绝条件、用户提示和记录方式 | 用户要求绕过审批或访问无权限合同 |
| 降级处理 | 降级触发条件和备用路径 | 检索证据不足时转为人工知识库工单 |

### 54.2.3 把评估写成证据，不写成口号

“准确率提升”“效率提升”只有在评估口径清楚时才有意义。案例至少要说明样本来自哪里，标注或复核由谁完成，失败类型如何分类，以及哪些指标只是过程指标。对 Agent 案例来说，最终答案准确率只是其中一项，还要检查工具调用是否正确、权限是否被遵守、人工确认是否按预期触发、失败时是否留下可复盘证据。

*表54-5：案例评估指标与适用边界。来源：本书整理。*

| 指标 | 适合回答的问题 | 风险 |
| --- | --- | --- |
| 任务完成率 | 系统能否走完整个任务链路 | 容易掩盖高风险动作被人工修正 |
| 证据命中率 | 回答是否引用到正确文档或数据 | 不能证明结论推理正确 |
| 工具调用成功率 | Tool 参数和返回是否符合契约 | 不能说明业务结果被接受 |
| 人工接管率 | 系统在边界处是否知道停下 | 过低可能代表越权，过高可能代表不可用 |
| 复核通过率 | 结果是否能通过业务人员检查 | 依赖复核标准的一致性 |

评估段落还要避免把所有指标写成“越高越好”。人工接管率就是典型例子。对于高风险报价、法务审阅或财务披露场景，接管率过低未必是好事，可能说明系统越过了该停下的边界；对于低风险 FAQ，接管率过高则说明系统无法独立处理常见问题。任务完成率也一样，如果系统把本该拒绝的请求也算作完成，指标会很好看，但风险很高。案例写作中要说明指标的适用边界，不能把它们当作统一评分。

评估结果最好能对应到失败分类。比如 DataAgent 案例可以把失败分成语义层口径错误、SQL 生成错误、查询超时、证据解释错误和权限拒绝；客服 Agent 可以分成政策检索失败、订单工具失败、退款规则冲突和人工接管超时。失败分类比单一准确率更能帮助读者理解平台该如何改进，也能回扣第38章至第42章的观测、评测、成本和 SLO 章节。

如果案例只有小样本，不要强行写成“上线效果”。可以写“在这组脱敏样本中，系统暴露出三类主要失败”，也可以写“当前材料只能支撑离线质量判断，尚不能支撑业务收益结论”。这种写法比编造 ROI 更可信，也更接近参考书的工程写作风格：先把条件和证据说清楚，再给出有限结论。

## 54.3 与前文的对应关系

*表54-6：各类案例与前文重点章节及关键工程问题的对应关系。来源：本书整理。*

| 案例类型 | 重点章节 | 关键问题 |
| --- | --- | --- |
| 企业知识助手 | 第20章、第21章、第27章、第40章 | 知识更新、证据引用、记忆边界和回答评估 |
| 工单/客服 Agent | 第8章、第23章、第30章、第51章 | 结构化输出、工具调用、人工接管和内容安全 |
| 研发 Agent | 第22章、第23章、第28章、第39章 | 长任务执行、代码工具权限、多 Agent 协作和评测 |
| 运维 Agent | 第38章、第42章、第45章、第46章 | Trace、SLO、网关治理和 GitOps 变更边界 |

表 54-6 的用途是防止案例与正文脱节，而不是要求每个案例覆盖所有章节。工单 Agent 若缺少结构化输出和人工接管，便很难支撑第8章和第30章的主张；运维 Agent 若缺少 Trace、SLO 和变更边界，也只能算工具演示，不能算生产案例。

### 54.3.1 用能力地图筛掉“只有功能没有平台”的案例

一个案例能否进入本书，还要看它是否反映了平台能力。如果材料只说明“调用大模型生成了一个结果”，却看不出 Runtime、Registry、Gateway、Memory、Eval、Trace 或 Guardrails 的作用，那么它更适合放在博客或产品介绍中，而不是放进本书正文。

审稿时可以使用一个简单判断：案例中至少应有三类平台能力真实出现，并且每一类都有证据。知识助手案例通常需要 RAG、Memory 和 Eval；工单 Agent 通常需要 Registry、Human-in-the-loop 和 Guardrails；DataAgent 案例通常需要语义层、sql_executor、Trace 和评估样本。

### 54.3.2 案例材料与工程证据的对应关系

第一版已经有 mini-platform 和测试用例。后续补写案例时，应优先选择能和现有工程路径对应的场景，而不是另起一套无法验证的系统。案例不一定把所有代码写进正文，但至少要说明每类材料落到哪类工程证据上。

*表54-7：案例材料与 mini-platform 最小对应关系。来源：本书整理。*

| 案例材料 | mini-platform 对应能力 | 正文应说明 |
| --- | --- | --- |
| 任务状态流转 | Runtime / Run 状态 | 状态如何从输入推进到完成或失败 |
| 工具调用记录 | Tool Registry / executor | 工具契约、权限和失败返回 |
| 指标口径与查询 | semantic layer / sql_executor | 口径绑定、SQL 校验和只读限制 |
| 证据引用 | RAG / retrieval | 文档版本、引用片段和证据覆盖 |
| 评估样本 | eval / benchmark | 样本来源、标签和失败分类 |

## 54.4 案例材料发布门槛

后续版本补写案例时，可以使用匿名化复合场景，但要明确这是工程常见场景而非真实公司故事。正文不得写入客户名称、生产数据、内部域名、真实工单、真实账号或未经验证的收益数字。若案例需要展示界面、日志或报表，应使用脱敏样例，并把图表来源写入图题。

补写流程建议分为素材登记、证据复核和正文改写。素材登记阶段，作者提交任务说明、脱敏样本、系统拓扑、风险边界和评估材料，编辑只判断材料是否完整，不先润色。证据复核阶段，审稿者检查材料是否支撑正文中的每一个结论；凡是涉及收益、准确率、时延、成本和上线规模的句子，都要有来源，没有来源的内容只能写成“待验证假设”或删除。通过这两步后，材料才进入正式案例写作。写作时优先保留因果链和失败边界，少写产品话术。

*表54-8：案例补写流程与退出条件。来源：本书整理。*

| 阶段 | 输入 | 产出 | 退出条件 |
| --- | --- | --- | --- |
| 素材登记 | 任务说明、样本、拓扑、截图 | 素材清单 | 材料缺关键链路时退回 |
| 证据复核 | 素材清单、评估记录 | 可写结论列表 | 结论无证据时删除或降级 |
| 正文改写 | 可写结论、章节映射 | 案例正文 | 通过 tone、图表、引用检查 |
| 发布复核 | 正文、图片、数据来源 | 合入版本 | 无敏感信息、无夸大收益 |

补写过程中，编辑应保留退回意见。案例被退回通常说明材料还缺关键证据，不等于写作失败。缺少工具调用记录，就不能写“系统能安全调用业务系统”；缺少评测样本，就不能写“质量达到可用水平”；缺少人工确认记录，就不能写“风险动作已被控制”。把退回原因记录下来，可以反向推动材料补齐，也能避免不同作者重复犯同一种问题。

图表补写也要有证据约束。案例中的流程图、状态机、工具拓扑和评测表都应来自脱敏材料或可解释的合成样例。图题要说明来源，正文引用要能对应到图表编号。若图中展示的是通用模式，可以写“来源：本书整理”；若图中来自真实项目脱敏，应写清楚“来源：脱敏案例改绘”。第一版暂不补具体案例图，是因为缺少足够可公开的业务证据，而不是因为案例章节不需要图。

后续版本如果要加入真实案例，建议先从单个场景写到可复审，再考虑扩展数量。一本书里三个证据扎实的案例，价值高于十个看似完整但无法复盘的故事。案例章节要让读者看到企业级 Agent 如何在数据、工具、权限、评估和组织责任之间落地，而不是证明所有行业都能用 Agent。

## 本章小结

本章目前只保留案例写作框架，不提供未经验证的业务案例。后续补写案例时，应先能复原任务、链路、证据和边界，再把平台能力放进去解释。案例写作不能用故事包装功能清单，也不能用虚构公司、截图或收益数字填补材料缺口。

脱敏范围要覆盖系统名称、数据样例、日志、工具参数、截图元数据和人员角色，替换公司名只是第一步。评估指标也要说明样本来源、复核口径和失败分类，不能把离线结果直接写成上线收益。只有这些材料齐备，案例才适合进入正文。
