## 1. DataAgent 能力边界与评测维度

本节先定义 DataAgent 的能力边界：理解业务问题、识别指标口径、定位数据源、生成查询、执行与纠错、分析结果、生成图表和报告、解释不确定性。

评测维度建议至少覆盖：意图理解、Schema Linking、SQL / Python 生成、执行成功率、答案正确性、分析洞察质量、可解释性、安全合规、时延和成本。重点强调不能只用“答案对不对”衡量 DataAgent。

## 2. Benchmark 任务空间设计

本节说明 benchmark 不是题库堆砌，而是任务空间设计。企业 DataAgent 的任务可以按问题类型、数据复杂度、分析动作、权限风险、交互轮次和产物形式分层。

建议给出任务矩阵：单表查询、多表 Join、指标口径解释、时间序列分析、异常归因、同比环比、分群对比、图表生成、报告摘要、长链路分析。

## 3. 企业数据分析场景抽象

本节把企业真实分析需求抽象成可复用场景。以财务、采购、销售、客服、供应链等场景为例，说明如何从业务流程中抽取指标、维度、事实表、口径规则和常见追问。

重点写清楚“场景抽象”的边界：不能泄露真实敏感数据，但要保留足够真实的数据形态、业务约束和分析复杂度。

## 4. QA 与任务集自动生成

本节介绍如何从语义层、元数据、历史报表、BI 看板和用户日志中生成 QA 与任务集。自动生成不是直接交给 LLM 造题，而是要有模板、约束、校验和人工抽检。

建议覆盖：问题模板、SQL 反推问题、指标口径扰动、多轮追问生成、难度分级、重复样本去重、泄漏检测和样本版本管理。

## 5. Ground Truth 与标注体系构建

本节说明 Ground Truth 的类型：精确答案、SQL 参考解、执行结果、语义等价答案、过程标签、人工评分 rubric。DataAgent 任务往往不存在唯一标准答案，因此要区分“硬标签”和“软标签”。

建议建立标注流程：样本入库、自动求解、专家审核、争议处理、版本冻结、回归集发布。对企业场景要强调权限、脱敏和审计。

## 6. 结果评测、过程评测与语义评测

本节把评测分成三层：结果评测看最终答案，过程评测看中间步骤，语义评测看解释和洞察是否合理。DataAgent 的关键在于过程可验证，否则最终自然语言答案很难直接判定。

建议覆盖指标：执行成功率、答案准确率、SQL 等价率、工具调用正确率、步骤收敛率、引用可信度、图表适配度、解释一致性、人工评分一致性。

## 7. 主流 DataAgent Benchmark 对比分析

本节对比公开 benchmark 的适用性。可以把 NL2SQL、表格问答、数据科学任务、通用 Agent 任务分开讨论，说明它们分别覆盖 DataAgent 的哪些能力，又缺少哪些企业约束。

建议对比维度：数据规模、Schema 复杂度、业务语义、是否多轮、是否需要工具执行、是否有权限约束、是否支持过程评测、是否适合持续回归。

## 8. 企业级持续评测平台建设

本节设计企业内部持续评测平台：样本库、执行器、Judge、指标看板、回归门禁、灰度对比、失败样本沉淀、版本追踪。

建议说明评测平台如何接入 CI/CD、模型路由、Prompt 版本、工具版本、语义层版本，以及如何与 Ch.38 的 trace 和 Ch.40 的线上反馈闭环。

## 本章小结

### 关键结论

1. DataAgent 评测不能只看最终答案，还要评估意图理解、数据定位、执行过程、解释质量和安全边界。
2. 企业 benchmark 的价值在于覆盖真实业务约束，而不是追求题目数量最大。
3. Ground Truth 可以是精确答案、参考 SQL、执行结果、过程标签或专家 rubric，不必强求单一形式。
4. 过程评测是 DataAgent 区别于普通问答评测的关键。
5. 持续评测平台要绑定模型、Prompt、工具、数据和语义层版本，才能支撑长期演进。

### 上线检查清单

- [ ] 是否为 DataAgent 定义了能力边界和分层评测指标？
- [ ] 是否有覆盖核心业务场景、难度分级和版本管理的 benchmark？
- [ ] 每次模型、Prompt、工具或语义层变更是否触发回归评测？
