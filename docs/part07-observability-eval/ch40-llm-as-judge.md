## 1. 线上反馈信号采集

本节定义线上可采集的反馈信号：显式点赞/点踩、用户追问、重新生成、复制下载、人工接管、工单升级、任务放弃、会话时长、结果采纳、业务系统后续动作。

重点说明反馈不是天然标签。用户点踩可能来自答案错误、表达不好、权限受限、等待太久或预期不一致，因此反馈信号必须结合 trace、任务类型和业务上下文解释。

## 2. 用户行为与业务指标建模

本节把用户行为转化为可观察的质量指标。Agent 平台既要看技术指标，也要看业务指标，如任务完成率、首次可用答案率、人工接管率、平均交互轮数、报表生成采纳率、决策动作转化率。

建议说明指标口径：哪些是领先指标，哪些是滞后指标；哪些适合作为告警，哪些适合作为产品优化参考。

## 3. LLM-as-Judge 评测框架

本节介绍 LLM-as-Judge 的基本框架：输入样本、候选答案、参考答案、评分 rubric、Judge 模型、评分输出、置信度和解释。

建议覆盖三种常见模式：单答案打分、成对比较、多维 rubric 评分。企业场景要强调 Judge 的输入脱敏、评分可追溯、模型版本固定和人工抽检。

## 4. 评测偏差与一致性控制

本节讨论 LLM-as-Judge 的风险：位置偏差、长度偏差、风格偏差、模型自偏好、rubric 模糊、领域知识不足、对事实错误不敏感。

基础控制方法包括：答案顺序随机化、成对评测、少量黄金样本校准、多 Judge 交叉、人工一致性抽检、rubric 版本管理和置信度阈值。

## 5. A/B 实验与能力验证

本节说明如何用线上实验验证能力变化。实验对象可以是模型、Prompt、工具策略、缓存策略、语义层版本或检索策略。

建议覆盖实验设计：流量分桶、用户隔离、样本量估算、护栏指标、主指标、实验时长、回滚条件。Agent 场景要注意多轮会话和长任务带来的归因复杂度。

## 6. 评测驱动的持续优化闭环

本节把线上反馈转化为持续优化流程：线上采样、自动评测、人工复核、失败聚类、修复方案、离线回归、灰度实验、指标观察、样本沉淀。

重点说明“评测驱动”不是为了追分，而是为了让每次优化都有证据、有边界、有回归保护。

## 本章小结

### 关键结论

1. 线上反馈信号需要和 trace、任务类型、业务上下文结合解释，不能直接等同于质量标签。
2. LLM-as-Judge 适合补充语义评测，但必须有 rubric、校准样本和一致性控制。
3. A/B 实验要同时设置能力指标和护栏指标，避免质量提升以成本、延迟或安全为代价。
4. 在线评测的价值在于发现离线 benchmark 覆盖不到的真实使用问题。
5. 持续优化闭环要把线上失败样本重新喂回离线评测和回归集。

### 上线检查清单

- [ ] 是否区分显式反馈、隐式行为和业务结果三类信号？
- [ ] LLM-as-Judge 是否有固定 rubric、版本管理和人工抽检？
- [ ] 每次线上实验是否设置了回滚条件和护栏指标？
