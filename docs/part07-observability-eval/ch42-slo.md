## 1. Agent 服务等级目标设计

本节说明 Agent SLO 的特殊性：除了可用性和延迟，还要考虑任务成功率、答案质量、工具成功率、人工接管率、成本上限和安全拦截。

建议区分不同场景的 SLO：交互式问答、长任务分析、后台批处理、审批流、报表生成。不同场景不应套用同一套延迟目标。

## 2. 延迟、成功率与质量指标

本节定义核心指标：首 token 延迟、总完成时长、模型调用延迟、工具调用延迟、任务成功率、步骤成功率、重试率、超时率、降级率、质量评分、人工接管率。

重点说明指标之间的关系：降低延迟可能损害质量，提高重试次数可能提高成功率但推高成本，过强限流可能保护系统但影响用户体验。

## 3. 限流、熔断与降级策略

本节介绍保护系统的三类策略。限流控制进入系统的请求，熔断阻断不健康下游，降级提供可接受的替代能力。

Agent 场景的降级方式包括：切换小模型、减少上下文、关闭非关键工具、返回异步任务、转人工处理、只给解释不执行动作、只读不写。

## 4. 长任务可靠执行机制

本节关注数分钟到数小时的 Agent 任务。长任务需要检查点、幂等工具调用、断点恢复、任务队列、心跳、超时控制、人工审批挂起和结果通知。

建议强调 Runtime 不能把长任务绑定在单个 HTTP 连接上。一次任务应有独立 `run_id`、可恢复状态和可查询进度。

## 5. 容量规划与弹性伸缩

本节说明 Agent 平台的容量规划对象：模型服务、LLM 网关、工具执行器、向量库、数据库、队列、回放存储和评测任务。

建议覆盖：流量峰值、并发任务数、平均步骤数、模型 token 吞吐、工具调用放大系数、缓存命中率、GPU / CPU / IO 资源、租户配额和弹性伸缩策略。

## 6. 企业级 Agent 平台稳定性建设

本节把 SLO、观测、成本和评测整合为稳定性工程。稳定性不仅是“不宕机”，还包括质量不退化、成本不失控、风险可拦截、事故可复盘。

建议给出稳定性机制：错误预算、值班告警、事故分级、变更冻结、灰度发布、回滚预案、压测演练、故障注入和复盘制度。

## 本章小结

### 关键结论

1. Agent SLO 要同时覆盖延迟、成功率、质量、成本和安全，不能只看服务可用性。
2. 限流、熔断和降级要按任务风险和业务价值分层设计。
3. 长任务必须具备检查点、幂等、断点恢复和进度查询能力。
4. 容量规划要考虑 Agent 的步骤放大效应，而不是只看入口 QPS。
5. 企业级稳定性建设需要把观测、评测、成本和 SLO 放在同一个运营闭环中。
