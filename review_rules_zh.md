# 《企业级 Agent 平台工程》中文第一版 Revising Agent 修订规则

本文件给后续 revising agent 使用。它不是“只 review 不动手”的评审清单，而是一套直接修订书稿、目录、图片引用、代码说明、CI 和网页电子书配置的执行规程。目标是把《企业级 Agent 平台工程：从数据智能底座到 AI 原生业务系统》的中文第一版修到“读者一眼看上去像一本正式技术书”。

参考基准：

- 已出版参考书：`datascale-ai/data_engineering_book`
- 参考图片目录：`https://github.com/datascale-ai/data_engineering_book/tree/main/docs/images/part1`
- 本地参考副本：`/Users/zhongyiliu/Desktop/data_agent/data_engineering_book/repo`
- 目标仓库：`/Users/zhongyiliu/Desktop/data_agent/enterprise_agent_book/enterprise_agent_platform_book`

参考书提供的是出版质量标尺：中文主线清晰、目录稳定、卷前导读能说明阅读路径、MkDocs 站点像电子书、图片按 Part 汇总、图表编号和正文引用稳定、图片可读且有预算、项目代码有 smoke test、验证 workflow 和发布 workflow 分离、问题和修订都能定位到文件行号。

如果本地参考副本和 GitHub `main` 不一致，以远端 `main` 为准；尤其是图片格式、图题、来源和 Alt text 写法，要优先参考远端已发布内容。

## 1. Agent 使命

revising agent 的任务是直接把仓库修到更接近中文第一版发布状态，而不是只输出问题。发现问题后，默认下一步是改文件、跑验证、留下修订日志；只有需要外部事实、真实资产、作者取舍或用户授权时，才把问题留到剩余清单。

必须执行：

1. 读取当前仓库和参考书关键结构。
2. 识别 P0/P1/P2/P3 问题。
3. 对 P0/P1 默认直接修改，除非修改需要外部事实、真实图片资产、作者判断或用户授权。
4. 对 P2 能顺手修的直接修；无法顺手修的记录在修订日志。
5. 每轮修订后运行相应验证命令。
6. 输出“改了什么、还剩什么、如何复审”。

不要只说：

- “建议优化”
- “需要补充”
- “应当统一”
- “图片需要检查”

要改成：

- 直接统一术语、目录、标题、图号、表号、链接、占位符。
- 直接补齐可从上下文推导出的导读、章节收束、图片说明、表格说明、上线检查项。
- 直接修正明显错误的路径、导航、仓库链接、MkDocs 配置和 CI 配置。
- 直接删除或移出发布导航中的空壳章、临时副本和未完成草稿；如果不能删除正文文件，也至少从发布导航移除。

单轮输出必须包含：

1. 真实文件修改，不得只有建议文本。
2. 验证命令和结果。
3. 剩余 P0/P1 清单及不能直接修的原因。
4. 下一轮最小可闭环修订范围。

## 2. 工作边界

revising agent 可以修改：

| 类型 | 可修改内容 |
| --- | --- |
| 正文 | `README.md`、`docs/**/*.md`、`PLAN.md`、`glossary/terms.md` |
| 导航与发布 | `mkdocs.yml`、`.github/workflows/**`、requirements 文件、站点样式脚本 |
| 质量脚本 | `scripts/**` 中用于检查目录、图片、语气、敏感信息、项目 smoke test 的脚本 |
| 代码说明 | `mini-platform/**` 的 README、示例配置、测试入口；必要时可修明显 bug |
| 图片引用 | Markdown 图片路径、图号、图题、来源、Alt text、读图说明、未引用图片清单 |
| 图片资产 | 按 Part 汇总、重命名、移动、压缩、替换已有图片；生成或重绘图片前需要确认工具和来源 |

必须谨慎：

1. 不编造技术事实、真实公司案例、法规条文、论文结论或工具能力。
2. 不把未实现能力写成已实现能力。
3. 不删除用户有意保留的内容，除非它明显是临时副本、空壳或发布污染。
4. 不引入真实密钥、邮箱、内网地址、客户数据、生产表名。
5. 不做大规模重构，除非它直接服务第一版发布。
6. 不提交或推送，除非用户明确要求。

如果无法直接补写，应采取可发布处理：

- 空壳章节：优先补成可读正式章；若缺少足够内容，移出第一版导航，并在修订日志记录。
- 缺失图片：优先修路径；没有可用资产时，移除引用或替换为可渲染 Mermaid/SVG 草图，并标明需要后续美化。
- 缺失项目代码：优先改正文为“规划能力”；如果章节核心依赖该项目，移出第一版核心目录或补最小 smoke 示例。

## 4. 优先级与动作

| 等级 | 含义 | Agent 默认动作 |
| --- | --- | --- |
| P0 | 阻断第一版 | 立即修；无法修则从发布导航移除或降级为草稿，并记录剩余阻断。 |
| P1 | 第一版前必须修 | 本轮尽量直接修；无法修则记录 owner/原因/下一步。 |
| P2 | 统稿问题 | 能顺手修就修；否则记录。 |
| P3 | 增强项 | 不影响本轮主线，除非成本很低。 |

P0 典型问题和直接动作：

| 问题 | 直接修订动作 |
| --- | --- |
| README/首页仍写 `v0.1 骨架`、`待主笔认领` | 改成当前真实版本说明，删除招募式措辞。 |
| `mkdocs.yml` 仍是 `your-org` | 改成真实仓库 URL 和发布地址；不确定发布地址时写当前 GitHub Pages 目标并标注 TODO 不进正文。 |
| 章节全是 TODO/空壳 | 补写正式章；不能补时从 nav 移出。 |
| 导航引用不存在文件 | 修路径或移除导航项。 |
| 图片链接缺失 | 修路径、移动图片或移除失效引用。 |
| 正文引用不存在代码目录 | 补最小目录/README/检查脚本，或把正文改为规划能力。 |
| 构建失败 | 优先修构建错误，再继续内容修订。 |
| 敏感信息 | 立即脱敏。 |

## 6. 仓库卫生与目录修订

直接修订规则：

1. 删除或移出发布路径中的临时副本、带空格编号副本、重复章稿，例如 `ch50 2.md`。
2. 所有正式发布章节必须在 `mkdocs.yml` 中出现；所有 nav 路径必须存在。
3. 首页、README、Part index 不得出现内部招募语气。
4. 每个 Part index 要说明本 Part 的位置、章节关系、建议读法。
5. 附录不能是空壳；空壳附录要补最小可用内容或移出第一版导航。
6. 文件名稳定、无空格、无副本后缀。
7. 中文第一版导航要统一：不要只在 Part I 混入英文导航，而其他 Part 是中文单语。

参考书结构可作为目标：

- `docs/zh/` 是中文主线。
- `docs/images/partN/` 是按 Part 汇总的共享图片资产区，例如参考书 Part I 使用 `docs/images/part1/`。
- README 有在线阅读、全书结构、项目结构、本地运行和验证命令。
- MkDocs 有书籍化导航、图片放大、footer、右侧目录跟随。

当前仓库可以分阶段迁移，不要求一轮完成目录重构；但 revising agent 应优先消除“明显半成品观感”。

图片目录迁移优先级：

1. 新增或重命名的中文第一版图片，优先放入 `docs/images/partN/`，不要继续扩大章节内 `images/`、`docs/images/ch/` 或零散截图目录。
2. 已有章节内图片可分批迁移；一旦迁移，必须同步修改 Markdown 引用并运行 MkDocs strict build。
3. 英文图片只放英文专用目录；中文第一版正文不得引用 `en/` 图片，除非该图本身无文字且标题、说明均已中文化。
4. 临时生成图、prompt 产物、未压缩原图不要直接进入发布图片池；源文件可以放到脚本或素材目录，但正文只引用最终导出图。

## 7. 正文修订规则

每章要修成正式技术书章节，而不是模板。

最低结构：

1. 章头结构固定为：`本章摘要 -> 关键词 -> 学习目标 -> 场景引入`，顺序不能打乱，不能再用引用块式“本章目标 / 前置阅读 / 估计阅读”。
2. 本章摘要：概括问题、方法、工程动作和章节承接，不写宣传语，不复述目录。
3. 关键词：使用 4-8 个短语级术语，用顿号分隔；不要把整句标题或半截句子放进关键词。
4. 学习目标：写成可验证动作，例如“能够解释边界”“能够设计接口”“能够识别失败模式”。
5. 场景引入：从读者能感知的真实业务场景或匿名化复合场景进入，不再使用“山岚集团”这类固定虚构公司。
6. 概念边界：仅在本章主题容易与相邻概念混淆，且这种混淆会影响后续判断时保留；不要为每章硬写“核心概念与边界”。
7. 架构/流程/状态/决策：按章节对象选择至少一种结构化表达；概念章可以用对比表或判断表，工程章可以用架构图、状态机、时序或接口表。
8. 实现、验证与恢复：只在本章有明确工程对象、运行链路、接口、配置或项目入口时保留；概念章、综述章和案例规则章不得虚构 mini-platform 路径、上线验收或失败恢复。
9. 图表说明：每张图/表都要有正文用途。
10. 本章小结：只收束本章关键结论和后续章节承接；上线检查、故障恢复或验收标准只在本章确有工程交付对象时保留，并放入相应实现小节。

### 7.0 旧模板删除规则

早期写作模板中的 `L1 概念 / L2 架构 / L3 工程实现` 只用于规划，不允许作为正式章节结构保留。revising agent 遇到下列标题或等价变体时，默认先判断是否删除，而不是润色：

| 旧模板小节 | 默认动作 | 允许保留的条件 | 推荐改法 |
| --- | --- | --- | --- |
| `核心概念与边界（含对比表）` | 删除或改名 | 本章确有多个容易混淆的概念，且后文会使用这些边界做决策 | 改成 `X.Y 术语、对象与相邻概念`，只保留必要定义和一张边界表 |
| `常见误区` | 删除或改名 | 误用会导致错误架构、错误选型、越权、成本失控或质量退化 | 改成 `X.Y 适用条件与反例`，按“现象 -> 原因 -> 工程动作”写 |
| `失败模式与恢复策略` | 删除或改名 | 本章主题有明确运行态、状态机、调用链或上线对象 | 改成贴合对象的标题，例如 `SQL 执行失败如何回退`、`审批超时如何恢复` |
| `生产化 checklist` / `上线检查清单` | 删除 | 本章没有可发布的服务、接口、配置、任务链路或项目代码 | 工程章改成对象化标题，如 `X.Y 模型路由上线验收`、`X.Y 采集链路接入条件`；概念章改成评估口径或直接删除 |
| `发布校验` / `异常定位` / `发布门禁` / `接入门禁` / `故障定界` | 改名或合并 | 本章确有工程交付对象、运行态或排障链路 | 改成 `上线验收`、`接入条件`、`事故复盘`、`事故定位`、`失效场景与修正`，并写出证据、指标和动作 |
| `踩坑记录` | 删除或改名 | 有真实工程现象、根因和修复动作，不是口语化经验罗列 | 改成对象化标题，如 `X.Y 查询执行事故复盘`、`X.Y GPU 调度事故复盘`，或并入相关实现小节 |
| `组件划分与接口契约` | 删除或拆分 | 本章确实定义平台组件和接口 | 工程章保留接口；选型章改成决策表；案例章改成证据链 |
| `mini-platform 关联` | 删除 | 本章没有真实仓库路径、配置或测试入口 | 只在路径存在且正文能解释运行前提时保留 |

删除判断：

1. 如果小节只是在复述标题，没有图表、代码、接口、指标、状态或证据，删除。
2. 如果小节内容可以自然并入上一节或下一节，合并后删除标题。
3. 如果小节为了满足“每章都要有”的模板而引入不存在的上线对象、项目路径或失败恢复，删除并改正文为概念判断或评估方法。
4. 如果删除会造成章节缺少必要承接，改成贴合本章对象的标题，而不是保留旧模板名。
5. 不得为了全书统一而让所有章节都出现同一组小节。统一的是章头、编号、图表、引用和写作口径，不是内容骨架。

按章节类型选择结构：

| 章节类型 | 应保留的结构 | 应删除的结构 |
| --- | --- | --- |
| 概念/综述章 | 定义、边界、架构地图、决策标准、章节承接 | 代码路径、上线清单、故障恢复、伪项目闭环 |
| 选型/评估章 | 场景分型、指标、对比表、决策流程、验证方法 | 接口契约、生产 checklist、空泛误区 |
| 工程实现章 | 输入输出、接口/配置、状态机、代码路径、验收和恢复 | 与实现无关的概念铺陈、泛泛价值说明 |
| 治理/组织章 | 责任边界、流程、证据、准入/复审、审计口径 | mini-platform 路径、技术接口、上线故障清单 |
| 案例/案例规则章 | 案例准入、脱敏、证据链、评估口径、平台化收束 | 虚构公司故事、编造收益、强行项目闭环 |

可直接改的写作问题：

- 删除空泛形容词和宣传腔。
- 把长段拆成“问题 -> 判断 -> 工程动作”。
- 把“我们将介绍”改成直接陈述。
- 把弱建议改成可执行清单。
- 保留并自然嵌入表号、图号。
- 删除“山岚集团”式虚构公司设定，改成具体行业、角色、系统和任务场景，例如“零售企业的经营分析团队”“制造企业的质检知识库”“银行客服质检团队”。
- 统一 Agent、Workflow、RAG、Copilot、DataAgent、Semantic Layer、NL2SQL、MCP、Guardrails、Memory 等术语。

不能直接编造的内容：

- 真实 benchmark 数字。
- 具体法规适用结论。
- 外部厂商能力声明。
- 尚未存在的 mini-platform 实现细节。
- 未提供来源的真实案例。

遇到上述内容，要改成“工程判断框架”或“示例字段”，不要编造成事实。

### 7.1 写作风格与学术表达规范

中文第一版的行文要学习参考书的组织形式：先交代问题，再拆解概念和工程约束，随后给出表格、图示、代码或检查项，最后收束到后续章节。revising agent 要直接改稿，不只标注“AI 感强”。

标题规范：

1. 章节标题使用稳定层级：`# 第X章 标题` 或 `# 第X章：标题` 二选一后全书统一；不要混用营销式标题、疑问句标题和口号式标题。
2. 二级标题用 `## X.Y 标题`，三级标题用 `### X.Y.Z 标题`；除“本章摘要”“关键词”“学习目标”“场景引入”“本章小结”“参考文献”外，不新增无编号大标题。
3. 标题应表达本节处理的工程问题或概念边界，例如“向量库在企业平台中的职责”，不要写“为什么它很重要”“深入理解核心能力”这类泛标题。
4. 同一章内标题长度和语法结构要相近；不要连续出现多个“从 A 到 B”“如何……”开头的标题。
5. 标题里中英文术语要克制，中文能准确表达时优先中文；英文缩写首次出现时在正文解释，不把标题变成缩写堆叠。

编号规范：

1. 章内小节编号、图号、表号、代码清单编号必须互相对应；修改标题或移动段落时，同步修改正文引用。
2. 正文引用图表用 `图 5-1`、`表 5-1`；图题/表题行用 `图5-1：`、`表5-1：`。
3. 列表编号只用于步骤、阶段、检查项；并列概念优先用表格或无序列表，避免长串“第一、第二、第三”。
4. 同一节内不要让多个编号体系同时竞争，例如“第一阶段 / Step 1 / P0 / L1”混用；确需并存时先定义它们的层级关系。
5. 章节承接引用要具体到章或节；如果无法确认章节号，改成概念引用，不能随意编章节号。

用词规范：

1. 使用技术书面语，避免宣传腔和短视频式表达，例如“颠覆”“秒懂”“史诗级”“一招解决”“开箱即用”“大杀器”。
2. 避免口语化、情绪化和夸张比喻，例如“踩坑”“翻车”“拉垮”“玄学”“炼丹”“核爆”“杀疯了”。
3. 禁止高频 AI 套话，例如“在当今快速发展的时代”“至关重要”“显著提升”“全面赋能”“无缝集成”“保驾护航”“不仅……而且……”。
4. 少用空泛判断，多写可验证对象：把“系统要更智能”改成“系统需要记录任务输入、工具调用、证据来源和人工接管状态”。
5. 避免把规划写成事实：未实现能力写“设计目标”“规划能力”“示例接口”，已验证能力才写“已实现”“可运行”“通过检查”。
6. 保留必要术语，但要给出工程口径，例如“可观测性”要落到 trace、metric、log、事件模型和告警阈值。

句式规范：

1. 避免连续段落都用“本章首先……随后……最后……”；摘要可以使用一次，正文要改成自然承接。
2. 避免机械的“不是 A，而是 B”反复出现；需要对比时优先用表格、边界定义或工程取舍说明。
3. 每段只处理一个判断，长段拆成“现象 -> 原因 -> 工程动作”。
4. 句子要有主语和动作，少写抽象名词串联，例如“能力建设推进落地闭环”。
5. 避免同一页重复使用“因此、同时、此外、进一步、最终”开头；修订时要替换成具体承接。
6. 不写自我介绍式文本，例如“本文将深入探讨”“接下来让我们看看”；改成直接陈述章节内容和读者可执行动作。

案例与场景规范：

1. 不再使用“山岚集团”作为统一虚构公司。读者难以建立真实感，且容易让全书像模板化案例。
2. 场景引入要直接写行业、角色、系统和问题，例如“某零售企业的经营分析团队在周会前需要解释毛利率下滑原因”。
3. 可以使用匿名化复合案例，但必须说明它是“匿名化复合场景”或“工程常见场景”，不要写成真实公司故事。
4. 一个场景必须服务本章主题：模型章节讲模型选择和推理约束，数据章节讲数据链路，评估章节讲指标和复核，不要让案例只做装饰。
5. 案例中出现的字段、指标、阈值、表名必须是示例口径；涉及真实业务数字时要标明“示例值”，不要伪装成调研事实。

段落组织规范：

1. 每章开头必须保留“本章摘要、关键词、学习目标、场景引入”，并与参考书的章头导读气质一致；摘要要概括问题、方法、工程动作和章节承接，不写宣传语。
2. 每个主要小节应至少包含一个可检查对象：表格、图、代码片段、流程、配置、指标口径或上线检查项。
3. 图表前要说明为什么需要看这张图/表，图表后要解释它如何影响工程决策。
4. 代码示例前要说明运行前提，代码后要说明输出、失败模式和生产化边界。
5. “本章小结”不要复述目录，要总结本章建立的判断框架、交付物和下一章承接。

去 AI 感硬规则：

1. 删除模板化开场、空泛价值判断、堆叠形容词和泛化排比。
2. 删除没有信息增量的句子，例如“这一点非常关键”“具有重要意义”“能够提供有力支撑”。
3. 同一页出现两次以上的固定句式必须改写。
4. 连续三段如果都是“概念解释 -> 重要性 -> 建议”，必须重组为“场景 -> 决策 -> 实施约束”。
5. 修订后朗读应像技术作者在解释工程问题，而不是像模型在扩写提纲。

## 8. 全书逻辑修订

修订时要维护全书主线：

1. Part I 定义 Agent、平台边界和 AI 原生业务系统。
2. Part II 解释模型选型、本地推理、推理优化、结构化输出和能力定制。
3. Part III 解释数据采集、湖仓、OLAP、实时、编排质量、元数据/血缘/指标。
4. Part IV 解释嵌入、向量库、文档解析、RAG 和知识工程。
5. Part V 解释 Runtime、Tool Registry、MCP、Planner、Workflow、Memory、多 Agent、协议、HITL、框架对标。
6. Part VI 用 DataAgent 把语义层、NL2SQL、Text-to-Pandas、可视化报告和生态对标串起来。
7. Part VII-X 收束到可观测、评估、成本、部署、前端、安全、合规和组织。
8. Part XI 案例必须回扣前面平台能力。

直接修订动作：

- 后文引用前文但章节号不对：修章节号或改成无编号引用。
- 术语首次出现无解释：补一句简明定义。
- 同一概念重复讲：保留新增视角，删除重复铺陈。
- 规划能力误写成已实现：改成规划能力，或补实现和测试。
- 图表编号不连续：统一图号、表号和正文引用。

## 9. 代码与项目修订

代码示例必须可信。

直接修订规则：

1. 正文引用的路径不存在：先查仓库是否有等价路径；有则改引用，无则改成规划能力或补最小文件。
2. 代码块无语言标记：补语言标记。
3. 命令缺上下文：补 `cd` 路径或前置依赖。
4. 示例含密钥、邮箱、内网地址、真实客户数据：脱敏。
5. 章节声称“项目演练”但无项目目录：补最小 README/run.py/test，或将章节改成“设计样例”并移出实战项目承诺。
6. `scripts/check_all.sh` 未覆盖新规则：补脚本或在 README 中明确验证命令。

参考书的项目 smoke test 只做轻量验证也可以接受：检查 README、环境文件、核心脚本、Python 编译或最小运行入口。当前仓库可采用同等策略。

## 10. 图片与图表修订

图片是第一版观感的关键，revising agent 不要只说“图片要检查”，要直接处理能处理的问题。

### 10.1 图片目录与命名

参考书的 Part I 图片集中在 `docs/images/part1/`，正文从 `docs/zh/part1/*.md` 以 `../../images/part1/<filename>` 引用。目标仓库应按同一思想收敛：图片资产按 Part 汇总，正文只保留稳定相对路径。

直接动作：

1. 新图和重命名图统一进入 `docs/images/partN/`，`N` 使用发布目录中的 Part 序号，不加前导零，例如 `part1`、`part2`、`part10`。
2. 当前章节仍在 `docs/part02-model-inference/ch05.md` 这类目录时，引用路径应写成 `../images/part2/<filename>`；如果后续迁入 `docs/zh/part2/ch05.md`，引用路径改成 `../../images/part2/<filename>`。
3. 不再新增 `docs/images/ch/`、`docs/partXX/images/`、`docs/partXX/zh/images/` 这类分散目录；已有目录只作为迁移来源。
4. 同一 Part 内不得出现同名不同图、同图多副本、`copy`、`副本`、`final-final`、带空格文件名。
5. 文件扩展名必须与真实格式一致；能用 SVG 表达的架构图、流程图、状态机、矩阵图优先 SVG，其次 PNG/WebP；照片和真实截图优先 WebP/JPEG/PNG。

命名规则：

1. 新增或重命名图片优先使用 ASCII 稳定名：`ch<章号>_fig<序号>_<英文短语>.<ext>`，例如 `ch05_fig01_model_matrix.svg`。
2. 参考书历史图片中存在 `data_quality_gates_1775835548587.svg` 这类语义名加生成时间戳，也存在 `图1_1_数据质量对比.png` 这类中文名。迁移时可以保留已稳定引用的旧名，但不要新增只有时间戳、无章节号、无语义的文件名。
3. 中文文件名只允许作为历史兼容；新文件优先 ASCII，避免不同操作系统、CI、URL 编码和发布环境出现问题。
4. 截图文件名要包含来源系统和用途，例如 `ch47_fig02_modelstudio_agent_config.webp`，不得使用 `screenshot1.png`、`image.png`。
5. 图号变更时，文件名、Markdown alt、图题、正文引用必须一起改；不能只改正文编号。

### 10.2 图片引用

直接动作：

1. 图片路径错：修 Markdown 路径。
2. 图片存在但未引用：判断是否应加入正文；无用则列入清理。
3. 图片缺失：查相邻目录和同名文件；找到则移动或改引用；找不到则移除引用或用 Mermaid/SVG 临时替代。
4. 中文页面引用英文图：改为中文图；没有中文图则记录为待生成，但先保证页面不混乱。
5. 英文页面引用中文图：若不处理英文正文，也至少记录污染，不让它影响中文第一版结论。
6. 图片引用必须使用相对路径，不使用 GitHub raw URL、绝对本机路径或站外临时链接。
7. 一张图只在一个规范位置维护；多个章节需要复用时，仍引用同一个 `docs/images/partN/` 文件，不复制图片。

### 10.3 图号、图题、来源、Alt text 与读图说明

直接动作：

1. 每张图必须有图号、图题、来源、Alt text、读图说明。
2. alt 文本为空、只有“图”、只有文件名，或与图题无关：改成描述性 alt。
3. 图号跳号/重号：按章重排并同步正文引用、文件名和图题。
4. 图片前后缺解释：补“这张图解决什么判断问题”。
5. 图号不要删；要让句子更顺，而不是让编号消失。
6. 图题不得只写在图片 alt 里；图片下一行必须有独立图题说明。
7. 正文第一次提到图片时使用 `图 5-1` 这种正文引用写法；图片标题行使用 `图5-1：`，与参考书一致。

推荐格式：

```markdown
![图5-1：企业模型矩阵与运行时路由，展示任务画像、治理策略与模型池之间的选择关系](../images/part2/ch05_fig01_model_matrix.svg)

*图5-1：企业模型矩阵与运行时路由。来源：本书自绘。该图展示任务画像、治理策略和模型池如何通过模型网关完成统一路由；Alt text：企业模型矩阵与运行时路由，展示任务画像、治理策略与模型池之间的选择关系。*
```

图表编号规则：

1. 图号、表号按章节独立编号：第 5 章第一张图是 `图5-1`，第二张是 `图5-2`；第 5 章第一张表是 `表5-1`。
2. 正文引用图表时写 `图 5-1`、`表 5-1`，正文中的空格用于中文阅读；图题和表题行写 `图5-1：`、`表5-1：`。
3. 不使用全书连续编号，不使用 `图 1`、`图一`、`Figure 5-1`、`Fig.`。
4. 同一图表不能同时出现 `图 5-1` 和 `图5-1` 两种标题行；标题行按无空格格式统一。
5. 如果引用外部资料，来源要写在图题/表题里；如果是本书整理或自绘，写 `来源：本书整理` 或 `来源：本书自绘`。
6. 表题放在表格前一行，格式为斜体：`*表5-1：标题。来源：本书整理，基于……归纳。*`。
7. 图题放在图片后一行，格式为斜体；表题和图题都不要用二级标题或粗体标题。
8. 每个图表前后至少有一处正文解释其用途，不允许图表孤立出现。

### 10.4 图片内容质量

直接动作：

1. 明显乱码、伪文字、箭头断裂、节点重叠：替换、重绘或移出第一版。
2. 截图含账号、密钥、真实数据：打码或替换。
3. 位图过大：压缩或转 WebP/JPEG；保留清晰度。
4. 架构图是位图且文字不清：优先改成 SVG/Mermaid。
5. 装饰图不能解释工程问题：删除或换成流程/架构/界面图。

图片硬标准：

- 单张位图建议不超过 5 MB；第一版新增图片优先控制在 1 MB 内。
- 全站图片总预算建议先按 400 MB 控制；单个 Part 图片预算应单独汇总。
- 中文图片的主标题、节点、坐标轴、图例必须可读。
- 移动端缩放后仍能理解主线。
- AI 生成图不能有伪文字和逻辑错线。
- SVG 必须能在 MkDocs 页面正常渲染；含中文时要检查字体 fallback，不要导出不可编辑、不可搜索的大块位图 SVG。
- 真实产品截图必须去除账号、组织名、密钥、URL token、客户数据和内部域名。

### 10.5 图片 Prompt 和源文件

直接动作：

1. 正文中的“配图 Prompt”“生成提示词”“Atlas prompt”删除。
2. 生成脚本可以留在 `scripts/`，但不能让读者在正文看到制作过程。
3. 如果有 Mermaid/Excalidraw/Figma/PPT 源文件，尽量保留源文件路径说明。
4. 没有源文件的大图，至少保证导出文件命名可追踪章节。
5. 图片源文件不作为正文引用目标；正文只引用最终导出的 SVG/PNG/WebP/JPEG。

## 12. CI 与自动化修订

参考书有 `validate.yml` 和 `publish.yml` 分离。当前仓库可以逐步靠近。

优先修：

1. validate 负责构建、敏感信息、术语、图片、tone、项目 smoke test。
2. publish 只在 validate 成功后部署。
3. markdownlint/link-check 不能长期 warn-only；如果暂时 warn-only，要写清第一版前转强门禁的计划。
4. 把 docs 依赖放入 requirements 文件，避免本地和 CI 不一致。
5. 新增 `scripts/check_image_sizes.py`，如果不存在。
6. 新增或扩展图片 inventory 检查，覆盖图片目录是否按 `docs/images/partN/` 汇总、文件名是否稳定、Markdown 引用是否存在、图题是否包含来源和 Alt text。
7. 新增或扩展 `scripts/check_tone.py`，覆盖“山岚集团”式虚构公司、AI 套话、宣传腔、口语化词汇、重复句式和标题编号异常。
8. 新增 `scripts/run_all_project_smoke_tests.py` 或 mini-platform 等价检查。

不要为了 CI 通过删除有价值内容。修根因。

## 13. 修订流程

每轮 revising agent 应按这个顺序做：

1. 记录仓库状态。
2. 跑占位符、导航、图片、构建扫描。
3. 列出本轮要直接落地的 P0/P1 修订目标。
4. 先修仓库卫生、构建、导航和首页。
5. 再修空壳章、Part index、附录。
6. 再修图表、代码路径、术语一致性。
7. 最后补 CI 自动化和网页体验。
8. 运行验证。
9. 输出修订日志。

不要一次性改全书所有章节。优先选一个可闭环范围，例如：

- 首页 + MkDocs + README
- Part VI 全部空壳章
- 图片资产专项
- mini-platform 示例路径专项
- 附录专项
