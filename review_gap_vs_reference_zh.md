# 对标参考书的差距清单与修订规则（中文第一版）

> 用途：本文件是一份**对比型 review / revise 规则**，给后续 revising agent 使用。
> 它把本书（`enterprise_agent_platform_book`）与参考书
> `datascale-ai/data_engineering_book` 的当前中文版本对比，指出仍需修改的结构、语言、内容密度和出版工程差距。
>
> 本文件不替代 `review_rules_zh.md`。`review_rules_zh.md` 规定怎样改，本文件规定“对标出版书还差什么、先改哪里”。

---

## 0. 当前结论

本书已经完成了几项过去缺失的基础工作：中文章节统一到 `docs/part*/ch/`，英文章节统一到 `docs/part*/en/`，图片按 `docs/images/partN/ch` 与 `docs/images/partN/en` 分语言归档，英文版也已经进入 MkDocs 的 `English Edition` 导航。因此，旧版清单中“Part I 混入 en”“中文 nav 不得引用 en”“图片必须按参考书平铺到 partN”这几条已经不再适用。

当前真正影响出版感的不是目录，而是三件事：

1. 正文论述密度不足。章节行数看起来接近参考书，但中文字符数明显偏低，很多篇幅由表格、清单和代码块撑起来。
2. 章节节奏不均衡。Part I 的前三章过长，Part XI 原本过短，Part IV 与 Part X 若干章仍偏薄。
3. 语言气质偏平台设计文档。参考书常用“问题复盘 -> 原因解释 -> 工程方法 -> 验证”的推进方式，本书更常用“平台位置 -> 职责表 -> 检查项”的说明方式。

## 2. 篇幅和密度规则

### 2.1 健康区间

正文章节建议落在以下区间：

- 中文字符数：4500-15000。
- 行数：300-750。
- 表格：优先控制在 4-10 张。超过 15 张时必须检查是否用表格替代了论证。
- 代码块：每个代码块前后都要解释运行前提、输出含义和失败边界。

这些不是机械字数要求。短章可以成立，但必须是数据集说明、附录说明或明确的索引型章节；长章也可以成立，但需要清楚的章节节奏和足够正文密度。

### 2.2 当前优先级

优先扩写或重写：

- Part XI：Ch54、Ch55。它们原本只有 79/88 行，已在本轮扩写为案例方法论和复审机制章节，后续还应继续补一个可运行 mini-platform 走读。
- Part IV：Ch18、Ch19、Ch20、Ch21。每章 214-242 行，虽然图表完整，但正文论证仍偏薄。
- Part X：Ch51-Ch53。安全、合规、组织章节不应只列框架，应补失败案例、责任边界和复审口径。

优先压缩或拆分：

- Ch01、Ch02、Ch03。前三章分别约 834、840、758 行，承担了过多概念定义和平台判断。建议保留开篇叙事，压缩重复边界表，把后文能展开的内容还给对应章节。

## 3. 语言和风格规则

### 3.1 少写“平台”，多写具体模块和动作

本书“平台”出现频率远高于参考书。这符合主题，但过量使用会让句子变空。修改时优先把抽象平台句落到具体模块、字段、状态和动作。

不推荐：

> 平台需要具备统一治理能力。

推荐：

> Registry 在工具注册时记录权限域、输入 schema 和审计字段；Runtime 执行前按租户策略拦截高风险调用。

### 3.2 少用表格替代段落

表格只能压缩比较关系，不能替代因果论证。后续修订时，每张表必须满足：

1. 表前有一段说明为什么需要这张表。
2. 表后有一段解释这张表带来的判断。
3. 表格内容不是正文句子的简单拆列。
4. 检查项只保留可执行动作，删除“建立机制”“完善治理”这类虚词。

### 3.3 四件套保留，但必须去模板感

本书保留“本章摘要 / 关键词 / 学习目标 / 场景引入”。但每章四件套都要像人工写作，而不是批量生成。

- 摘要：直接陈述本章要解决的工程问题，不写“本章先...再...最后...”。
- 关键词：4-8 个短语级术语，不放整句标题，不放带冒号的标题。
- 学习目标：每条必须含本章特有对象，例如 schema-linking、Run 状态、reranker、Guardrails 策略等。
- 场景引入：必须引入任务冲突、失败后果或决策压力，不要只写“本章讨论什么”。

### 3.4 参考书的可学习写法

参考书更常采用这种推进：

1. 具体问题或事故。
2. 问题为何难以靠直觉解决。
3. 方法、架构或代码。
4. 评估、风险和工程边界。

本书后续修订应优先把“平台位置说明”改成“问题链”。比如 NL2SQL 章节不只说它在 Planner 里，还要先说明单轮 prompt-to-SQL 在真实 schema、权限和资源限制下为什么失败。

## 4. 内容补强规则

### 4.1 短章补正文，不补清单

短章不能靠新增“上线检查清单”“常见失败模式”凑长。优先补：

- 一个具体任务或失败场景。
- 错误路径为什么自然发生。
- 正确工程路径怎样一步步建立。
- 哪些模块负责，输入输出是什么。
- 评估或复审证据是什么。

### 4.2 长章先压重复概念

Ch01-Ch03 不建议继续扩写。应删减重复定义、重复表格和提前展开的后文章节内容。原则：

- Ch01 只讲 Agent 的本质、边界和任务闭环。
- Ch02 只讲企业级 Agent 平台的边界、共享能力和治理理由。
- Ch03 只讲 AI 原生业务系统怎样改变企业软件，不重复 RAG/Copilot/Workflow/Agent 的基础区分。

### 4.3 案例章节要从方法论走向最小走读

Ch54/Ch55 现在已从占位说明扩成方法论章节，但仍缺参考书那种“项目互相印证”的闭环。下一步应补一个不编造公司的 mini-platform 最小走读，例如：

- DataAgent 经营分析：语义层 -> NL2SQL -> sql_executor -> Trace -> 评估样本。
- 工单/客服 Agent：结构化输出 -> Tool Registry -> 人工接管 -> Guardrails -> 复审记录。

这类走读不需要声称真实上线，但要能用现有 mini-platform 和测试支撑。

## 5. 图表和图片规则

图片目录策略已经确定：

- 中文正文引用 `../../images/partN/ch/<file>`。
- 英文正文引用 `../../images/partN/en/<file>`。
- 不再要求对齐参考书的图片平铺目录。

仍需改进的是图题质量：

- Alt text 不得简单重复标题，要描述图里的模块、箭头、状态和关系。
- 来源为“本书自绘”可以保留，但引用或改编外部资料时必须写明依据。
- 图后正文要解释图的判断，不要让图独立悬浮。

## 6. 工程检查缺口

当前已有：

- `check_chapter_template.py`
- `check_figures_tables.py`
- `check_markdown_tables.py`
- `check_citation_style.py`
- `check_tone.py`
- `check_image_sizes.py`
- `check_translation_alignment.py`
- `check_chapter_density.py`

仍建议补：

1. `check_references.py`：正文引用与章末参考文献互校。
2. `check_xref.py`：校验“第N章”、Markdown 内链和章节号是否真实存在。
3. `check_quant.py`：标记百分比、金额、QPS、延迟、规模数字，要求来源或示例声明。
4. 图题 Alt text 重复检测：当 `图N-M：标题` 与 `Alt text：标题` 完全相同，应告警。

## 7. 后续执行顺序

1. 跑 `python3 scripts/check_chapter_density.py`，按 thin-prose 和 table-heavy 排序。
2. 继续扩写 Part IV 与 Part X 的短章，补问题链和工程证据。
3. 压缩 Ch01-Ch03，删除提前展开和重复表格。
4. 补一个 mini-platform 最小走读，优先 DataAgent 或工单/客服 Agent。
5. 补参考文献、交叉引用、量化数据和 Alt text 检查脚本。
6. 每轮修订后跑：

```bash
bash scripts/check_all.sh
python3 scripts/check_translation_alignment.py
.venv/bin/mkdocs build --strict --clean --site-dir /tmp/enterprise-agent-book-site
```
